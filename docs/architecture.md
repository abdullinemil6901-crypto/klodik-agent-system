# Архитектура: как устроены слои

Дополнение к [README](../README.md): README даёт схему, этот файл — устройство каждого слоя и решения, которые за ним стоят.

## 1. Слой памяти

Проблема: каждая новая сессия с ассистентом начинается с нуля, качество ответа зависит от того, что успел пересказать.

Решение — файловая память из двух частей:

- **CLAUDE.md** в корне рабочей папки. Claude Code читает его автоматически в начале сессии. Внутри — правила работы и карта: где профиль, где журналы, где проекты. Пересказывать контекст не нужно ни разу.
- **Obsidian vault** — база знаний обычными markdown-файлами:

```
vault/
├── MEMORY.md                  ← индекс, с него начинает любой агент
├── 00 — Профиль/              ← кто пользователь, стиль ответов, обязательные правила
├── 01 — Память агента/        ← журнал фидбэка, статусы, выводы сессий
├── 02 — Wishlist и репозитории/
├── 03 — Skills и MCP/         ← каталоги подключённых инструментов
└── 04 — Проекты/              ← по файлу/папке на проект
```

Ключевое: агент не только читает память, но и **пишет в неё**. После нетривиальной сессии — запись в журнал фидбэка: что сработало, что нет, какой вывод. Ошибка, разобранная один раз, не повторяется в следующих сессиях — система накапливает опыт, как сотрудник, а не как чат.

Обезличенный скелет vault лежит в [`vault-template/`](../vault-template/) — можно взять и заполнить под себя.

Почему markdown-файлы, а не векторная БД или облачная память: объём памяти на одного пользователя маленький (десятки файлов), обычный поиск по файлам покрывает всё, содержимое читается и правится руками в любом редакторе. Специализированное хранилище станет оправданным, когда vault перерастёт файловый поиск — не раньше.

## 2. Роутинг: single vs multi-agent

Multi-agent — не дефолт, а инструмент под конкретный класс задач. Замер на живых прогонах: конвейер из нескольких агентов заметно поднимает качество, но стоит ~15× токенов.

Правило роутинга:

| Тип задачи | Пример | Режим |
|---|---|---|
| Линейная | подгонка резюме, рефакторинг, письмо | single-agent |
| Параллелизуемая | поиск под разными углами, аудит, скоринг | multi-agent конвейер |

На линейной задаче второй агент не добавляет качества — шаги зависят друг от друга, параллелить нечего, токены сгорают впустую. На параллелизуемой — углы поиска независимы, и adversarial-проверка отсеивает то, что одиночный агент выдал бы за факт.

## 3. Multi-agent конвейер: finder → verifier → synthesizer

- **Finder-агенты (×N)** ищут параллельно, каждый под своим углом — например, «технологии» и «рыночный спрос». Углы формулируются так, чтобы пересечение источников было минимальным.
- **Verifier** — скептик с презумпцией: *данные неверны, пока не подтверждены вторым независимым источником*. Проверяет даты и цифры, ищет конфликт интересов (площадка хвалит собственный рынок — не источник), режет хайп без подтверждений.
- **Synthesizer** собирает выжившее в результат под контекст пользователя из слоя памяти: не «список фактов», а «что это значит для тебя и что с этим делать».

Честная оговорка: взаимопроверка агентов одной модели — не панацея, слепые зоны у них общие. Поэтому verifier обязан опираться на **независимые источники и инструменты**, а не на «мнение» второго агента, и на высоких ставках в петле остаётся человек.

Разбор живого прогона с конкретными вердиктами verifier — в [demo_verifier_run.md](demo_verifier_run.md).

## 4. Рутины и доставка

- Плановые прогоны — OS cron + headless-запуск Claude Code; промты прогонов версионируются в [routines/](../routines/README.md), правка файла меняет поведение рутины без правки crontab.
- Отчёты пишутся в vault — Obsidian синхронизирует их на любое устройство.
- Telegram-доставка готова в обе стороны ([спека](telegram_bot.md)): push дайджестов из vault ([telegram_digest.py](../scripts/telegram_digest.py)) и интерактивный бот с кнопками решений, пишущий выбор человека обратно в журнал vault ([telegram_bot.py](../scripts/telegram_bot.py)). Принцип: vault — источник истины, Telegram — интерфейс. Дальше по MVP-пути: голос → разбивка на роли.

Экономика рутин: частота подбирается под лимиты подписки, а не наоборот. Multi-agent прогон радара 3×/день — это ~9 полных конвейеров в сутки; старт с 1×/день, повышение — после замера расхода.
