# Telegram-доставка: спецификация модуля

Модуль слоя «доставка и рутины» (см. [architecture.md §4](architecture.md)). MVP реализован: push-доставка дайджестов из vault в Telegram одним скриптом. Интерактив (кнопки, голос, роли) — фаза 2, спроектирована ниже, не реализована.

## Решение и его обоснование

Один stdlib-скрипт [`scripts/telegram_digest.py`](../scripts/telegram_digest.py), zero dependencies. Альтернативы отклонены на дизайн-ревью: n8n-workflow требует 24/7-рантайм и ручную чистку экспортов ради одного POST; бот-фреймворк (aiogram) — venv и зависимости ради экономии ~50 строк при переходе к фазе 2. Это то же правило, что и «markdown вместо векторной БД»: специализированный инструмент — когда задача его перерастёт, не раньше.

Принцип: **vault — источник истины, Telegram — только уведомление.** Потерянный пуш не теряет данные — дайджест лежит в vault; поэтому очередь доставки не нужна.

## Схема потока

```
cron-прогон конвейера
  └─ пишет дайджест → vault/04 — Проекты/Поиск работы/дайджесты/digest_latest.md
  └─ последним шагом вызывает:
       python3 scripts/telegram_digest.py --latest "<папка дайджестов>" \
           --env-file ~/.klodik/telegram.env
            └─ markdown → Telegram-HTML → sendMessage → чат пользователя
```

Вход — либо явный путь к файлу, либо `--latest DIR` (самый свежий `.md` по mtime; `digest_format.md` и журнал сбоев пропускаются). Основной контракт — фиксированное имя `digest_latest.md`, которое прогон перезаписывает: смена схемы имён не ломает доставку.

## Контракт дайджеста

Формат зафиксирован в [vault-template/04 — Проекты/Поиск работы/дайджесты/digest_format.md](../vault-template/04%20—%20Проекты/Поиск%20работы/дайджесты/digest_format.md): YAML front-matter с машиночитаемыми данными (`run_id`, `stats`, `items[]` с `id`/`title`/`url`/`fit`/`zone`) плюс человекочитаемое тело 5–10 строк. Доставка читает из front-matter только `run_id` (ключ идемпотентности); `items[]` — задел фазы 2: кнопки соберутся из данных, конвейер трогать не придётся.

Поддерживаемая разметка тела: `[текст](url)`, `**жирный**`, `# заголовок`. Выбран HTML parse_mode: экранировать нужно 3 символа (`&`, `<`, `>`) против 18 у MarkdownV2. Превью ссылок отключены — 5–10 ссылок на вакансии иначе превращают чат в простыню.

## Поведение при сбоях

| Ситуация | Поведение |
|---|---|
| Сеть/таймаут/5xx | до `TELEGRAM_RETRIES` попыток, экспоненциальный бэкофф (при дефолтных трёх — паузы 2 и 4 с) |
| HTTP 429 | сон на `retry_after` из ответа, попытка не сгорает (максимум 5 ожиданий) |
| HTTP 400 (кривая разметка) | одноразовый фолбэк: тот же текст plain, ссылки как `текст (url)` |
| Сообщение > 4096 симв. | разбиение по границам строк |
| Повторный запуск | идемпотентность: ключ `run_id` (фолбэк — sha256 тела) в `~/.local/state/klodik/`; прогресс пишется атомарно после каждой части, поэтому повтор дозаливает недоставленное без дублей; всё доставлено → «уже отправлено», код 0 |
| Финальный сбой | строка в `delivery_failures.md` рядом с дайджестом — следующая сессия агента увидит недоставленный пуш через слой памяти, не читая cron-логи |

Коды выхода: `0` — доставлено или уже было; `1` — ошибка конфигурации; `2` — доставка не удалась; `3` — дайджест не найден/пуст. `--dry-run` печатает готовое сообщение и ключ, не требует токена.

## Безопасность

- Секреты только через окружение или `--env-file` с файлом **вне репозитория** (`~/.klodik/telegram.env`, `chmod 600`). В репо — только [`scripts/telegram.env.example`](../scripts/telegram.env.example) с пустыми значениями; `.gitignore` режет `*.env`.
- Токен входит в URL запроса, поэтому тексты ошибок и логи собираются без URL: наружу уходят только код HTTP, `description` из ответа Bot API и класс сетевой ошибки. Проверено тестом.
- Приватностный риск: рабочие дайджесты содержат названия компаний и ссылки на вакансии. Любой пример в репозитории и vault-template — только плейсхолдерный.

## Экономика

Доставка — 0 токенов: скрипт работает вне LLM. Стоимость прогона конвейера не меняется.

## Фаза 2: кнопки решений (спроектировано, не реализовано)

Следующий шаг MVP-пути «один бот и одна роль → голос → разбивка на роли»:

- Дайджест приходит с inline-кнопками по серой зоне: «в работу» / «пропустить». Кнопки собираются из `items[]` front-matter; `callback_data` = стабильный ID вакансии (`JV-0001`) — лимит Telegram 64 байта, поэтому ID короткие.
- Обработчик callback — отдельный процесс-поллер, **co-located с vault и с правом записи**: решение пишется в журнал откликов, человек-в-петле работает прямо из Telegram.
- Обработчик обязан проверять `chat_id`/`user_id` отправителя — бот публично доступен по имени, входящие без авторизации не исполняются.
- Отправитель не переписывается: `send_message(text, reply_markup=None)` уже принимает клавиатуру.
